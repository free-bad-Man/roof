# Agents Map — Крымская Кровельная

Статус: карта зон ответственности

## 1. Принцип работы агентов
У проекта один архитектурный центр и несколько узких зон ответственности. Каждый агент должен:
- читать `docs/project/master-plan.md` перед началом работы;
- работать только в своей области;
- фиксировать изменения в своих файлах;
- не переписывать чужой source of truth без архитектурного решения.

## 2. Владельцы основных проектных файлов
| Файл | Владелец | Что фиксирует | Кто не должен переписывать |
|---|---|---|---|
| `docs/project/master-plan.md` | Архитектор проекта | рамку, инварианты, зависимости, архитектурные правила | все остальные без отдельного решения |
| `docs/project/agents-map.md` | Архитектор проекта | зоны ответственности и границы между агентами | все остальные |
| `docs/project/roadmap.md` | Архитектор проекта / release lead | этапы, порядок запуска, состояние фазы | исполнители без согласования |
| `docs/project/decision-log.md` | Архитектор проекта | утверждённые решения и изменения инвариантов | все остальные |

## 3. Карта агентов по зонам ответственности

### 3.1. Агент-архитектор
**Роль:** держит общий каркас проекта.

**Отвечает за:**
- `docs/project/master-plan.md`
- `docs/project/agents-map.md`
- `docs/project/roadmap.md`
- `docs/project/decision-log.md`

**Решает:**
- что является source of truth;
- какие сущности вообще существуют в проекте;
- какие изменения допустимы, а какие размывают бренд и структуру.

**Не делает:**
- весь дизайн;
- полный копирайтинг сайта;
- полную ручную реализацию кода.

### 3.2. Агент по структуре сайта и контент-каркасу
**Роль:** раскладывает утверждённую архитектуру по страницам и шаблонам контента.

**Работает на основе:**
- `docs/project/master-plan.md`
- `docs/project/roadmap.md`
- handoff / SEO-дерева / готовых текстов

**Может создавать и вести:**
- `docs/site/pages-map.md`
- `docs/site/templates.md`
- `docs/site/internal-linking.md`

**Не меняет:**
- верхнеуровневую архитектуру сайта;
- список ключевых направлений;
- бренд-позиционирование.

### 3.3. Агент UX/UI
**Роль:** превращает утверждённую структуру в интерфейсную систему.

**Работает на основе:**
- `docs/project/master-plan.md`
- брендпак / брендбук
- структура блоков главной и услуг

**Может создавать и вести:**
- `docs/design/ui-rules.md`
- `docs/design/wireframes.md`
- `docs/design/components-map.md`

**Не меняет:**
- брендовые инварианты;
- название компании;
- состав страниц первой волны;
- архитектуру CRM и SEO.

### 3.4. Агент frontend
**Роль:** реализует шаблоны страниц, UI и клиентские сценарии в коде.

**Отвечает за области репозитория:**
- `app/` или `src/app/`
- `components/`
- `styles/`
- клиентские части форм и CTA

**Входы:**
- master-plan
- wireframes / UI-rules
- pages-map

**Не меняет:**
- бизнес-логику CRM;
- SEO-архитектуру;
- бренд-стратегию.

### 3.5. Агент backend / integration
**Роль:** серверная логика, формы, база данных, интеграции.

**Отвечает за области репозитория:**
- `prisma/`
- `lib/`
- `server/` или серверные route handlers
- интеграции с amoCRM / Метрикой / уведомлениями
- Docker / Nginx-конфигурации по согласованной схеме

**Может создавать и вести:**
- `docs/integrations/forms-crm-flow.md`
- `docs/integrations/deploy.md`
- `docs/integrations/env-map.md`

**Не меняет:**
- контентную структуру сайта;
- визуальную систему;
- верхний стек без решения архитектора.

### 3.6. Агент SEO
**Роль:** усиливает индексацию, коммерческие страницы, перелинковку и рост по Яндексу.

**Может создавать и вести:**
- `docs/seo/metadata-map.md`
- `docs/seo/indexation-plan.md`
- `docs/seo/content-plan.md`
- `docs/seo/geo-policy.md`

**Не меняет:**
- продуктовую архитектуру ради количества страниц;
- brand voice;
- CRM-сценарии.

### 3.7. Агент CRM / marketing ops
**Роль:** внедряет воронку, поля, автоматизации, шаблоны сообщений и контроль источников.

**Может создавать и вести:**
- `docs/crm/implementation-checklist.md`
- `docs/crm/forms-to-amocrm.md`
- `docs/crm/source-tracking.md`

**Не меняет:**
- архитектуру сайта;
- структуру релиза;
- дизайн интерфейсов сайта.

### 3.8. Агент QA / release
**Роль:** проверяет целостность перед выкладкой и соблюдение инвариантов.

**Может создавать и вести:**
- `docs/release/checklist.md`
- `docs/release/smoke-tests.md`

**Не меняет:**
- архитектурные решения;
- контентную стратегию;
- визуальную систему.

## 4. Матрица зависимостей без дублирования контекста
| Агент | Читает | Пишет | Передаёт дальше |
|---|---|---|---|
| Архитектор | стратегия, handoff, брендпак, CRM-ТЗ | project/*.md | всем |
| Структура сайта | master-plan, roadmap | docs/site/* | UX/UI, SEO, frontend |
| UX/UI | master-plan, брендовые источники, docs/site/* | docs/design/* | frontend |
| Frontend | master-plan, docs/design/*, docs/site/* | код интерфейса | backend, QA |
| Backend / integration | master-plan, CRM-ТЗ, docs/site/* | код интеграций, docs/integrations/* | QA, release |
| SEO | master-plan, handoff, docs/site/* | docs/seo/* | frontend, content |
| CRM / marketing ops | master-plan, CRM-ТЗ, docs/integrations/* | docs/crm/* | backend, QA |
| QA / release | всё утверждённое по своей проверке | docs/release/* | architect / deploy |

## 5. Правило эскалации
Агент обязан вернуть вопрос архитектору, если хочет:
- добавить новый раздел верхнего уровня;
- создать новую сущность контента;
- поменять git flow или деплой-путь;
- добавить сложную CRM-архитектуру;
- изменить позиционирование или бренд-подачу;
- развернуть гео-масштаб без подтверждённого спроса.

## 6. Минимальный порядок запуска следующими агентами
1. Архитектор фиксирует рамку.
2. Агент структуры сайта делает pages map и шаблоны.
3. UX/UI собирает wireframe-систему первой волны.
4. Frontend + backend собирают каркас и интеграции.
5. SEO и CRM подключаются на уровне наполнения и валидации.
6. QA выпускает проект в релизный чек-лист.
