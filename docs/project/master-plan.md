# Master Plan — Крымская Кровельная

Статус: основной проектный файл

## 1. Назначение документа
Этот файл — главный источник истины по архитектуре проекта сайта «Крымская Кровельная». Он фиксирует границы проекта, неизменяемые принципы, приоритеты запуска, состав обязательных систем и связи между рабочими потоками.

Документ не дублирует брендбук, handoff, SEO-тексты и CRM-ТЗ. Он задаёт рамку, по которой работают все следующие агенты и исполнители.

## 2. Контекст проекта
### Бренд
- Маркетинговое название: **Крымская Кровельная**
- Полное название: **Крымская кровельная компания**
- Подача бренда: взрослая, профильная, надёжная компания по кровле, а не универсальная стройка

### Формула проекта
- **Кровля — основа**
- **Гидроизоляция — вход**
- **Sinzatim — усиление экспертности**
- **Потолки — поток**

### Бизнес-ядро
- Основной профиль: ремонт, восстановление и гидроизоляция кровли
- Входной продукт: устранение протечек и гидроизоляция
- Поточное допнаправление: натяжные потолки

## 3. Инварианты проекта
Эти правила нельзя менять без отдельного решения в `docs/project/decision-log.md`.

1. Не пересобирать позиционирование с нуля.
2. Не уводить бренд в универсальную стройку.
3. Не раздувать сайт десятками слабых страниц.
4. Во всех публичных сценариях первым считывается кровля.
5. Все ключевые решения фиксируются в проектных файлах, а не в переписке.
6. Каждый агент работает в своей зоне ответственности и не переписывает чужой source of truth.
7. Продакшн-деплой идёт только из `main`.
8. Разработка ведётся в одном репозитории с ветками `dev` и `main`.

## 4. Source of truth по проекту
### Управляющие файлы проекта
- `docs/project/master-plan.md` — архитектурная рамка проекта
- `docs/project/agents-map.md` — зоны ответственности и правила передачи задач
- `docs/project/roadmap.md` — последовательность этапов запуска
- `docs/project/decision-log.md` — журнал утверждённых решений

### Внешние утверждённые источники
- Стратегия запуска — бизнес-логика, контур услуг, SEO-ядро, ограничения по раздуванию
- Handoff для нового чата — целевая архитектура сайта, структура URL, первая волна запуска, формы, CRM-связка
- Мини-брендпак / брендбук — правила имени, визуальной системы, логотипа, тональности
- ТЗ по amoCRM — финальная практическая конфигурация CRM и логика лид-обработки

Правило: если возникает конфликт между новым предложением и утверждёнными источниками, новое предложение не внедряется до фиксации решения в `decision-log.md`.

## 5. Архитектура продукта
### 5.1. Цель сайта
Собрать SEO-сильный сайт под Яндекс, который:
- удерживает кровлю как главный профиль;
- даёт понятную структуру услуг;
- собирает лиды через формы, расчёт и отправку фото объекта;
- поддерживает кейсы, статьи, локальность и доверие;
- связан с amoCRM без перегруза интерфейса и сценариев.

### 5.2. Верхний уровень структуры
Обязательное верхнее меню:
- Главная
- Кровельные работы
- Гидроизоляция
- Натяжные потолки
- Наши работы
- О компании
- Контакты
- Статьи

### 5.3. Коммерческое ядро
#### Кровля
- Ремонт кровли
- Восстановление кровли
- Гидроизоляция кровли
- Восстановление мягкой кровли
- Восстановление мягкой кровли по технологии Sinzatim
- Монтаж кровли
- Замена кровли
- Кровля под ключ

#### Гидроизоляция
- Устранение протечек
- Гидроизоляция балконов и террас
- Локальный ремонт и герметизация узлов

#### Потолки
- Натяжные потолки под ключ
- В квартиру
- В частный дом

### 5.4. Контентный слой
Обязательные типы контента:
- коммерческие страницы услуг;
- кейсы / наши работы;
- статьи по модели «проблема → решение»;
- контакты и география;
- FAQ на ключевых страницах.

### 5.5. Что сознательно не раздуваем
- десятки подуслуг ради количества;
- гео-страницы по всем городам × всем услугам;
- комплектующие и мелкие допродажи как отдельное ядро сайта;
- тяжёлые визуальные блоки без SEO и лидовой пользы.

## 6. Архитектура технологий
### 6.1. Базовый стек
- **Frontend / app layer:** Next.js + TypeScript
- **Backend / server logic:** Next.js server capabilities
- **Database:** PostgreSQL
- **ORM:** Prisma
- **Infra:** Docker + Nginx
- **Production:** VPS

### 6.2. Среды
- Локальная разработка — ноутбук
- Production — VPS
- Отдельный dev-стенд на VPS не создаём

### 6.3. Git flow
- `dev` — интеграционная ветка разработки
- `main` — ветка production
- Деплой в production — только из `main`

### 6.4. Технические требования проекта
Сайт должен быть:
- HTML-first;
- пригодным для обхода Яндексом;
- не-SPA без SSR;
- быстрым на мобильных;
- с чистыми URL;
- с корректными canonical, sitemap, robots, schema и хлебными крошками.

## 7. Архитектура лидогенерации
### 7.1. Обязательные точки захвата
- форма первого экрана;
- форма страницы услуги;
- форма «запросить замер / отправить фото»;
- кликабельный телефон;
- кнопки мессенджеров;
- мобильная липкая панель связи.

### 7.2. Базовый лидовый сценарий
1. Пользователь оставляет заявку или отправляет фото.
2. Заявка попадает в amoCRM.
3. Менеджер квалифицирует лид.
4. Назначается выезд или расчёт по фото.
5. Отправляется смета.
6. Сделка переводится в работу или в отказ с причиной.

### 7.3. Принцип форм
- мало форм;
- мало полей;
- скрытые поля для контекста страницы;
- без длинных квизов и лишней анкеты на входе.

## 8. Архитектура CRM
### 8.1. Роль amoCRM
amoCRM — рабочий пульт менеджера, а не сложная CRM-экосистема.

### 8.2. Минимальная конфигурация
- одна основная воронка;
- простые этапы;
- ограниченный набор полей;
- базовые автоматизации;
- контроль дублей;
- источники и каналы без раздувания.

### 8.3. Подключаемые каналы на старте
- Сайт
- Телфин
- Telegram
- VK
- Почта

MAX — только вторым этапом, если подтверждён практический смысл.

## 9. Архитектура бренда в интерфейсе
### Обязательные правила
- На первом экране и в маркетинговой подаче использовать **Крымская Кровельная**.
- В официальных блоках, документах и реквизитах использовать **Крымская кровельная компания**.
- Логотип не менять, окно и трубу не убирать.
- Визуальная система: светлый фон, графитовый текст, красный акцент.
- Сайт не должен выглядеть как пёстрая бытовая реклама.

## 11. Зависимости между потоками работ
### Последовательность
1. Архитектор фиксирует рамку, файлы и порядок работы.
2. UX/UI и контент-структура работают только внутри утверждённой архитектуры.
3. Frontend реализует шаблоны и UI по утверждённой структуре.
4. Backend / integration подключает формы, CRM, метрики и системные сценарии.
5. SEO-агент наполняет метаданные, перелинковку, индексацию и контентный ритм.
6. CRM / marketing ops запускает обработку лидов и контроль источников.
7. QA / release проверяет целостность перед выкладкой в `main`.

### Критичное правило
Ни один следующий агент не должен заново менять:
- позиционирование;
- ядро структуры сайта;
- стек;
- git flow;
- модель CRM;
- базовую бренд-подачу.

## 12. Правила изменений
Изменение считается архитектурным, если затрагивает хотя бы один пункт:
- новые разделы верхнего уровня;
- изменение бизнес-ядра услуг;
- смена стековых решений;
- изменение среды релиза;
- новая отдельная воронка или сложная CRM-логика;
- новая сущность контента, которая раздувает структуру.

Такие изменения сначала фиксируются в `decision-log.md`, и только после этого становятся рабочими.

## 13. Критерии готовности проекта к запуску
Проект считается готовым к первому релизу, если:
- опубликована первая волна страниц;
- формы передают лиды в CRM;
- работает базовая мобильная версия;
- robots, sitemap, canonical и метаданные настроены;
- контакты, география, цены и CTA едины по всему сайту;
- бренд считывается как профильная кровельная компания;
- production-сборка выкладывается из `main`.

## 14. Что не входит в роль архитектора
Архитектор не делает:
- полный дизайн всех экранов;
- написание всего контента сайта;
- микро-реализацию каждого компонента;
- операционное ведение рекламы;
- ручное внедрение CRM вместо интегратора.

Роль архитектора — удерживать каркас проекта и не допускать расползания решений.
