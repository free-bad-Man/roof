# Webhook rules

## Назначение документа

Зафиксировать правила обработки входящей заявки через уже существующий `/api/leads` и правила внутренней интеграции с amoCRM на MVP.

На этом этапе под `webhook` понимается серверное правило обработки входящего запроса из сайта и дальнейшего запроса в amoCRM.

---

## 1. Общие правила

1. Основной публичный endpoint остаётся `/api/leads`.
2. Новый публичный endpoint `/api/leads/site` не вводится как основной.
3. Модуль интеграции работает только как внутренняя библиотека.
4. Backend на MVP не дублирует тело заявки в собственной БД.
5. В CRM создаются / обновляются только согласованные сущности:
   - контакт;
   - сделка;
   - примечание;
   - задача.

---

## 2. HTTP-правила для `/api/leads`

### Допустимый метод
- `POST`

### Нежелательные методы
- `GET`
- `PUT`
- `PATCH`
- `DELETE`

Для них route должен возвращать `405 Method Not Allowed`.

### Форматы тела
Допустимые форматы на уровне route:
- `application/json`
- `application/x-www-form-urlencoded`
- `multipart/form-data` (если текущий frontend уже так отправляет)

После чтения тела route должен привести данные к общему объекту `SiteLeadPayload`.

---

## 3. Валидация до вызова amoCRM

Минимальные обязательные проверки:
- телефон присутствует;
- телефон после нормализации не пустой;
- строковые поля обрезаны по краям;
- пустые строки превращены в `undefined`.

Дополнительные безопасные проверки:
- `message` ограничивается по длине;
- `name`, `city`, `service` ограничиваются по длине;
- route не пропускает гигантские payload'ы в integration layer.

Если валидация не пройдена:
- route возвращает `400`;
- в amoCRM ничего не отправляется.

---

## 4. Правила нормализации

### Телефон
- удаляются лишние символы;
- сохраняется исходное значение для человека;
- для dedupe используется нормализованная версия.

### Услуга
- берётся из `service`, если она пришла;
- если `service` пустая, пробуем определить по `pagePath`;
- если услугу определить нельзя, поле можно не заполнять автоматически.

### Источник
- для сайта по умолчанию `Сайт`.

### Город
- если значение совпадает с утверждённым списком, отправляем его как есть;
- иначе ставим `Другое` или не заполняем, в зависимости от политики вызова.

---

## 5. Правила dedupe

### Основной ключ
- `phone`

### Вторичный ключ
- `email`

### Поведение
1. Есть совпадение контакта -> обновляем контакт.
2. Есть активная сделка в воронке `Продажи` -> обновляем её и добавляем note.
3. Активной сделки нет -> создаём новую сделку.

### Чего не делаем
- не создаём отдельную таблицу дублей в своей БД;
- не блокируем заявку только потому, что dedupe не сработал идеально;
- не теряем обращение из-за спорного совпадения.

---

## 6. Правила по задачам

Для нового лида может создаваться задача:
- текст: `Связаться с клиентом`;
- привязка: к сделке;
- срок: по умолчанию в ближайшее рабочее окно или через configurable delay.

Если обновляется уже активная сделка:
- повторную задачу создавать необязательно;
- по умолчанию модуль не спамит задачами при каждом дубле.

---

## 7. Правила по notes

В common note допустимо уносить:
- исходный текст обращения;
- `formName`;
- `pagePath`;
- `referrer`;
- UTM;
- служебный комментарий о том, что лид обновлён, а не создан.

В note не нужно дублировать весь JSON payload без необходимости.

---

## 8. Ошибки интеграции

### Что считается интеграционной ошибкой
- amoCRM API недоступен;
- авторизация невалидна;
- обязательные ID полей / статусов не переданы;
- API вернул неожиданный ответ.

### Правило MVP
При интеграционной ошибке:
- не пишем тело заявки в локальную БД;
- не создаём теневую очередь в БД;
- не молча глотаем ошибку.

Модуль должен:
- вернуть `fallback`-объект;
- записать ошибку в лог;
- сохранить в памяти ответа достаточно данных для ручного алерта со стороны route.

---

## 9. Правила безопасности

- секреты amoCRM читаются только из env;
- access token не должен уходить в response;
- note не должен содержать лишние приватные служебные поля;
- route не должен возвращать наружу внутренние ID полей и статусов, кроме случаев отладки на сервере.

---

## 10. Правила расширения после MVP

Позже можно добавлять:
- private retry-job;
- alert в Telegram/почту администратору;
- inbound webhook'и из amoCRM для служебной синхронизации.

Но текущий пакет специально не внедряет:
- новую публичную точку приёма заявок;
- локальную БД лидов;
- новые сущности CRM поверх уже утверждённых.
