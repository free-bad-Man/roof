# amoCRM mapping

## Контекст

Этот mapping собран под фактическую структуру аккаунта amoCRM проекта:
- одна воронка: `ПРОДАЖА`;
- короткие формы сайта;
- одна основная серверная точка приёма заявок: `/api/leads`;
- backend на MVP не дублирует лиды в собственной БД;
- модуль интеграции работает как внутренний сервисный слой, который можно импортировать в уже существующий `src/app/api/leads/route.ts`.

## Что делает модуль

Модуль принимает нормализованный payload от сайта, превращает его в значения для amoCRM и работает только с теми сущностями, которые уже согласованы для MVP:
- контакт;
- сделка;
- примечание к сделке;
- задача.

Новые публичные endpoint'ы, новые воронки и новые сущности не вводятся.

## 1. Воронка и статусы

### Воронка
- `ПРОДАЖА`

### Фактические этапы воронки
1. `Неразобранное`
2. `Новая заявка`
3. `Квалификация`
4. `Осмотр / расчёт`
5. `Смета / ждём решение`
6. `Договор / согласование`
7. `Авансирование`
8. `Оплата получена`
9. `В работе`
10. `Успешно реализовано`
11. `Закрыто и не реализовано / отказ`

### Статус по умолчанию для заявки с сайта
- `Новая заявка`

`/api/leads` в текущем прямом сценарии создания заявки использует только:
- `AMOCRM_PIPELINE_ID`;
- `AMOCRM_STATUS_NEW_ID`.

Логика создания лида не должна зависеть от ID остальных этапов. Остальные `AMOCRM_STATUS_*_ID` нужны как optional-настройки для документации, будущих автоматизаций, отчётности и безопасного расширения интеграции.

### Логика входящих каналов
- Заявки сайта создаются сразу в `Новая заявка`.
- `Неразобранное` используется для входящих каналов amoCRM: телефония, Telegram, VK, почта.
- Дальнейшее движение по этапам делает менеджер или будущие автоматизации amoCRM.

> Модуль интеграции не меняет фактическую воронку, не вводит промежуточные статусы и не переводит сделку дальше `Новая заявка` при создании заявки с сайта.

## 2. Входящие формы сайта

На старте модуль ориентирован на 3 утверждённых сценария формы.

### Форма 1. Главная
Ожидаемые поля:
- `name`
- `phone`
- `service`
- `message`

### Форма 2. Страница услуги
Ожидаемые поля:
- `name`
- `phone`
- `city`
- `message`
- `service` (скрытое поле, подставляется по странице)

### Форма 3. Замер / отправка фото
Ожидаемые поля:
- `name`
- `phone`
- `city`
- `message`

Дополнительно backend может передавать:
- `pagePath`
- `formName`
- `utm`
- `referrer`
- `objectType`
- `needsVisit`
- `email`
- `telegram`

## 3. Mapping полей сайта -> поля сделки amoCRM

| Вход сайта | Поле amoCRM | Тип | Правило |
|---|---|---|---|
| `service` / `pagePath` | `Услуга` | список | Нормализуется в одно из утверждённых значений CRM |
| `source` | `Источник` | список | Для форм сайта всегда `Сайт`, если не передано иное |
| `city` | `Город` | список | Нормализуется по списку городов; иначе `Другое` |
| `objectType` | `Тип объекта` | список | Если не пришло — не заполняется автоматически |
| `message` | `Что нужно сделать` | текст | Основное краткое описание задачи |
| `needsVisit` | `Нужен выезд` | да / нет | Проставляется только если явно вычислено на backend |
| `visitAt` | `Дата и время замера` | дата / время | На первом создании лида обычно пусто |
| `address` | `Адрес объекта` | текст | На первом создании лида обычно пусто |
| `estimateTotal` | `Сумма сметы` | число | На первом создании лида обычно пусто |
| `lossReason` | `Причина отказа` | список | На первом создании лида не заполняется |

## 4. Mapping полей сайта -> поля контакта amoCRM

| Вход сайта | Поле контакта amoCRM | Правило |
|---|---|---|
| `name` | `Имя` | Используется как имя контакта |
| `phone` | `Телефон` | Обязательное поле для создания / поиска дубля |
| `telegram` | `Telegram` | Заполняется, если поле есть и есть ID поля в config |
| `email` | `Email` | Заполняется, если пришёл email |

## 5. Нормализация услуг

Сайт может прислать длинное человекочитаемое название услуги. В CRM модуль приводит его к фиксированным значениям.

| Сайт / маршрут | Значение CRM `Услуга` |
|---|---|
| `Ремонт кровли` | `Ремонт кровли` |
| `Восстановление кровли` | `Восстановление кровли` |
| `Гидроизоляция кровли` | `Гидроизоляция кровли` |
| `Восстановление мягкой кровли` | `Мягкая кровля / Sinzatim` |
| `Восстановление мягкой кровли по технологии Sinzatim` | `Мягкая кровля / Sinzatim` |
| `Замена кровли` | `Замена кровли` |
| `Монтаж кровли` | `Монтаж кровли` |
| `Кровля под ключ` | `Кровля под ключ` |
| `Устранение протечек` | `Устранение протечки` |
| `Гидроизоляция балконов и террас` | `Балкон / терраса` |
| `Локальный ремонт и герметизация узлов` | `Герметизация узлов` |
| `Натяжные потолки` / `Натяжные потолки под ключ` | `Натяжные потолки` |

Если `service` не пришёл, модуль пробует восстановить его по `pagePath`.

## 7. Что уходит в примечание, а не в custom fields

Чтобы не плодить новые поля и не ломать уже утверждённую схему, дополнительные технические данные сохраняются в common note сделки:
- название формы;
- route страницы;
- referrer;
- UTM-метки;
- исходный текст заявки;
- служебные пометки backend.

Это позволяет не менять состав полей сделки и контакта.

## 8. Дубли и стратегия обновления

MVP-правило:
1. сначала ищем контакт по телефону;
2. если телефон не найден и есть email — ищем по email;
3. если контакт найден, обновляем контакт;
4. если у контакта найдена активная сделка в той же воронке, обновляем её и добавляем note;
5. если активной сделки нет — создаём новую сделку и привязываем к существующему контакту.

Если поиск активной сделки недоступен или API вернул нештатный ответ, модуль создаёт новую сделку для найденного контакта. Это безопаснее, чем терять обращение.

---

## 9. Шаблон названия сделки

Используется уже согласованный формат:
- `Услуга — Город — Имя`
- если имя пустое: `Услуга — Город — Новый клиент`
- если город пустой: `Услуга — Новый клиент`

Примеры:
- `Ремонт кровли — Симферополь — Андрей`
- `Устранение протечки — Ялта — Марина`
- `Натяжные потолки — Новый клиент`

## 10. Что нужно заполнить в amoCRM account перед подключением модуля

В `config.ts` модуль ждёт:
- ID воронки `ПРОДАЖА`;
- ID статуса `Новая заявка`;
- ID пользовательских полей сделки;
- ID пользовательских полей контакта;
- список закрывающих статусов, чтобы не обновлять завершённые сделки как активные.

На текущем этапе для создания заявки через `/api/leads` обязательны только `AMOCRM_PIPELINE_ID` и `AMOCRM_STATUS_NEW_ID`. Остальные ID этапов можно заполнить позже, когда появятся автоматизации или логика движения по воронке.
