# Form flow

## Цель

Зафиксировать MVP-поток заявок без изменения уже существующего frontend и без создания нового основного endpoint.

Главная связка остаётся такой:

`сайт -> /api/leads -> integration module -> amoCRM`

---

## 1. Что не меняется

На этом этапе не меняем:
- публичный endpoint `/api/leads`;
- frontend-маршруты;
- состав коротких форм сайта;
- состав утверждённых полей сделки и контакта;
- модель "без дубля контента в своей БД" на backend MVP.

Модуль подключается как внутренний слой, который потом импортируется в существующий `src/app/api/leads/route.ts`.

---

## 2. Поток по шагам

### Шаг 1. Пользователь отправляет форму
Форма на сайте отправляет POST в существующий endpoint:
- `POST /api/leads`

Тело запроса может отличаться по форме, но после нормализации должно сводиться к общему payload.

### Шаг 2. Route делает только транспортную работу
`src/app/api/leads/route.ts` в будущем должен:
- принять запрос;
- провалидировать минимально обязательные поля;
- привести вход к `SiteLeadPayload`;
- вызвать `submitSiteLead()` из integration module;
- вернуть результат клиенту.

### Шаг 3. Service слой нормализует payload
Модуль:
- чистит телефон;
- нормализует услугу;
- определяет источник (`Сайт`);
- определяет город;
- собирает заголовок сделки;
- формирует note из технических атрибутов запроса.

### Шаг 4. Service ищет дубль
Алгоритм:
1. по телефону;
2. затем по email, если есть;
3. затем пытается найти последнюю активную сделку контакта в воронке `Продажи`.

### Шаг 5. Модуль создаёт или обновляет сущности в amoCRM
Сценарии:
- нет контакта -> создаём контакт + создаём сделку;
- контакт есть, активной сделки нет -> обновляем контакт + создаём сделку;
- контакт есть, активная сделка есть -> обновляем контакт + обновляем сделку + добавляем note.

### Шаг 6. Ставится задача на первичную обработку
При новом лидe модуль может создать задачу `Связаться с клиентом`, если включён флаг `createInitialTask`.

### Шаг 7. Route возвращает короткий результат
На frontend желательно возвращать только безопасный результат:
- `ok: true/false`
- `mode: created | updated | fallback`
- `leadId`
- `contactId`
- `message`

Никаких деталей по внутренней конфигурации amoCRM во внешний ответ отдавать не нужно.

---

## 3. Поддерживаемые формы

### 3.1. Главная форма
Вход:
- имя;
- телефон;
- услуга;
- краткое описание задачи.

Результат:
- создаётся или обновляется лид в статусе `Новая заявка`;
- `Источник = Сайт`;
- `Услуга` берётся из формы.

### 3.2. Форма со страницы услуги
Вход:
- имя;
- телефон;
- город;
- комментарий;
- скрытое поле `service`.

Результат:
- в CRM уходит точная услуга;
- маршрут страницы и название формы уходят в note.

### 3.3. Форма «Замер / отправить фото»
Вход:
- имя;
- телефон;
- город;
- что нужно сделать.

Результат:
- услуга определяется по `service` или `pagePath`, если возможно;
- если услугу определить нельзя, используется безопасный fallback `Ремонт кровли` только если backend явно разрешил это правило;
- иначе поле `Услуга` остаётся пустым, а менеджер уточняет её в квалификации.

> По умолчанию в модуле агрессивный автоподбор услуги не включён.

---

## 4. Какие поля считаются обязательными для входа в `/api/leads`

Минимум для успешной отправки в CRM:
- `phone`

Желательно:
- `name`
- `service` или `pagePath`
- `message`

Если нет телефона, endpoint должен вернуть `400` до обращения в amoCRM.

---

## 5. Что НЕ должно происходить в route на MVP

`src/app/api/leads/route.ts` пока не должен:
- писать лиды в локальную БД;
- создавать второй публичный endpoint для сайта;
- раздувать payload новыми сущностями;
- решать бизнес-логику mapping внутри route;
- дублировать код нормализации, который уже есть в модуле.

Route должен быть тонким.

---

## 6. Где проходит ответственность

### Frontend
Отвечает за:
- отправку в `/api/leads`;
- корректное имя полей формы;
- передачу `pagePath`, `formName` и при наличии UTM;
- короткие формы без усложнения UX.

### Route `/api/leads`
Отвечает за:
- транспорт;
- минимальную валидацию;
- вызов integration module;
- формат внешнего ответа.

### Integration module
Отвечает за:
- нормализацию;
- mapping;
- dedupe;
- запросы к amoCRM;
- fallback при неуспешной интеграции.

### amoCRM
Отвечает за:
- хранение лида и контакта;
- задачи;
- переписку и дальнейшую ручную обработку.

---

## 7. Что происходит при ошибке amoCRM

Если amoCRM временно не отвечает или возвращает ошибку:
- route не пишет копию заявки в БД;
- integration module возвращает `fallback`-результат;
- backend пишет ошибку в лог;
- при необходимости route может отдать пользователю нейтральный ответ, а команде — алерт.

Это соответствует согласованной модели: на MVP backend не дублирует контент лидов в собственной БД.

---

## 8. Рекомендованный контракт для будущего route

```ts
import { submitSiteLead } from '@/lib/integrations/amocrm';

const result = await submitSiteLead({
  name,
  phone,
  email,
  telegram,
  service,
  city,
  objectType,
  message,
  pagePath,
  formName,
  referrer,
  utm,
});
```

Route при этом остаётся совместимым с существующим `/api/leads`.

---

## 9. Что можно делать следующим шагом после подключения

После того как docs и integration module будут добавлены в repo, следующий отдельный шаг:
- аккуратно подключить `submitSiteLead()` внутри `src/app/api/leads/route.ts`;
- не меняя URL endpoint;
- не меняя frontend-маршруты;
- не создавая новый основной endpoint `/api/leads/site`.
