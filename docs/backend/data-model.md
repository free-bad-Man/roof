# Data model — Крымская Кровельная

## Принципы

1. **Кровля остаётся ядром**, поэтому в модели есть явное поле `direction`, но нет абстрактного каталога “всех строительных услуг”.
2. **Контент не раздуваем**: отдельные сущности есть только для того, что реально заявлено в проекте — услуги, кейсы, статьи, заявки и SEO для статичных страниц.
3. **Формы не превращаем в таблицу-конструктор**: тип формы хранится как enum в заявке. Это дешевле в поддержке и не требует админки раньше времени.
4. **amoCRM не дублирует базу сайта**: локально храним только то, что нужно для захвата лида, дедупликации, аудита и повторной синхронизации.

Это соответствует зафиксированной логике проекта: кровля — основа, гидроизоляция — вход, Sinzatim — усиление экспертности, потолки — поток, а сайт строится на сильном коммерческом ядре без раздувания страниц. fileciteturn2file1 fileciteturn2file0

## Список сущностей

### 1) `Service`
Коммерческая услуга сайта.

Что покрывает:
- разделы и страницы услуг из утверждённого ядра;
- H1 / title / description;
- цену “от” и единицу цены;
- связь с кейсами, статьями и лидами;
- CRM-значение для поля «Услуга».

Почему отдельная сущность нужна:
- у проекта есть фиксированное коммерческое ядро услуг;
- формы услуг должны подставлять услугу автоматически;
- статьи и кейсы должны перелинковываться с конкретными услугами. fileciteturn2file1 fileciteturn2file0

### 2) `CaseStudy`
Кейс / работа / объект.

Что хранит:
- slug и path;
- SEO и публикационный статус;
- город, тип объекта, проблема, решение, результат;
- галерею и обложку;
- ссылку на основную услугу.

Почему нужна:
- кейсы входят в обязательный SEO-слой;
- на страницах услуг должны быть реальные объекты и 1–2 кейса. fileciteturn2file1

### 3) `Article`
Статья по модели “проблема → решение”.

Что хранит:
- slug и path;
- SEO и публикационный статус;
- обложку, excerpt, контент;
- связь с основной услугой.

Почему нужна:
- статьи входят в утверждённый SEO-контур и перелинковку. fileciteturn2file1 fileciteturn2file0

### 4) `Lead`
Единая запись входящего обращения с сайта.

Что хранит:
- источник;
- тип формы;
- услугу;
- город, тип объекта, текст задачи;
- признак выезда;
- вложения/фото в JSON;
- UTM и referrer;
- локальный статус;
- состояние синхронизации с amoCRM.

Почему нужна:
- CRM должна принимать заявки с сайта, не терять обращения и не быть перегруженной лишними статусами;
- в amoCRM зафиксированы конкретные поля сделки и контакта, и локальная база должна маппиться на них без промежуточной лишней сущности. fileciteturn2file2 fileciteturn2file0

### 5) `StaticPageSeo`
SEO-метаданные для статических маршрутов, которые не завязаны на отдельную контентную сущность.

Примеры:
- `/`
- `/o-kompanii/`
- `/kontakty/`
- `/krovelnye-raboty/`
- `/gidroizolyatsiya/`
- `/simferopol/`

Почему нужна:
- title / description / H1 обязательны для индексируемых страниц;
- не хочется тянуть универсальный page-builder только ради SEO. fileciteturn2file0

## Что сознательно **не** делаем

### Нет отдельной сущности `Form`
Форм на старте мало и они фиксированы:
- главная;
- услуга;
- замер / отправка фото. fileciteturn2file0

Отдельная таблица форм только усложнит проект. Для аналитики и маршрутизации достаточно `formType + source + pagePath + utm` внутри `Lead`.

### Нет отдельной сущности `MediaAsset`
Для первой версии галереи и вложения хранятся в `Json`.

Причина:
- сейчас нет задачи строить медиа-библиотеку и админку;
- этого хватает для связки с S3/объектным хранилищем или внешним CDN позднее.

### Нет отдельной таблицы синхронизаций / webhook-лога
Пока хватает полей в `Lead`:
- `amoLeadId`
- `amoContactId`
- `amoSyncStatus`
- `amoSyncError`
- `amoPayload`

Если позже понадобится полноценный аудит интеграции, можно добавить `CrmSyncLog`, но **не сейчас**.

## Кардинальности

- `Service 1 -> N CaseStudy`
- `Service 1 -> N Article`
- `Service 1 -> N Lead`
- `StaticPageSeo` живёт отдельно и адресуется по `path`

## Поля `Lead` и их соответствие amoCRM

| Lead | amoCRM |
|---|---|
| `name`, `phone`, `telegram`, `email` | Контакт |
| `source` | Поле сделки «Источник» |
| `serviceId` / `crmOfferValue` | Поле сделки «Услуга» |
| `city` | Поле сделки «Город» |
| `objectType` | Поле сделки «Тип объекта» |
| `message` | Поле сделки «Что нужно сделать / краткое описание задачи» |
| `needsVisit` | Поле сделки «Нужен выезд» |
| `address` | Поле сделки «Адрес объекта» |
| локальная синхронизация | `amoLeadId`, `amoContactId`, `amoSyncStatus` |

Набор полей намеренно совпадает с практической конфигурацией amoCRM: одна воронка, минимум полей, без лишней бюрократии. fileciteturn2file2

## Публикация контента

Для `Service`, `CaseStudy`, `Article` используется общий принцип:
- `DRAFT` — черновик;
- `PUBLISHED` — доступно сайту и sitemap;
- `ARCHIVED` — снято с выдачи, но не удалено.

Это достаточно для первой версии и не требует отдельной редакторской системы.

## SEO-стратегия в модели

### Для услуг / кейсов / статей
SEO хранится внутри сущности:
- `h1`
- `seoTitle`
- `seoDescription`
- `canonicalPath`
- `noIndex`

### Для статических страниц
SEO хранится в `StaticPageSeo`.

Так мы избегаем полиморфной таблицы “seo для всего подряд”, но покрываем все утверждённые требования:
- уникальный title;
- уникальный description;
- один H1;
- canonical. fileciteturn2file0

## Индексы

В схеме добавлены только полезные индексы:
- публикационные по контенту;
- `slug`, `path` как `unique`;
- индексы по `Lead.status`, `Lead.source`, `Lead.formType`, `Lead.phone`, `Lead.email`, `Lead.amoLeadId`.

Этого достаточно для:
- выборки опубликованного контента;
- фильтрации лидов;
- дедупликации и повторной синхронизации.

## Расширения, которые можно добавить позже

Без ломки текущей модели позже можно добавить:
- `GeoPage` для действительно уникальных гео-страниц;
- `Review` для отзывов;
- `CrmSyncLog` для расширенного аудита интеграции;
- `MediaAsset` при появлении полноценной медиа-библиотеки.

Но для текущего этапа это лишнее.
