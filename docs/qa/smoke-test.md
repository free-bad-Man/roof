# Smoke test

## Когда запускать
- после merge в `main`
- сразу после production deploy на VPS
- после hotfix, затрагивающего формы, SEO, routing или deploy

## Критерий результата
Если падает хотя бы один пункт из раздела A или B, релиз считается неуспешным до исправления.

## A. Доступность и базовый HTTP
1. Открыть главную по HTTPS.
   - Ожидаемо: `200 OK`, без mixed content, без redirect loop.
2. Проверить, что сайт открывается без `www` и по каноничной версии домена.
3. Проверить, что HTTP -> HTTPS отрабатывает корректно.
4. Проверить, что сервер не отдаёт `502/504` на главной и ключевых маршрутах.

## B. Ключевые маршруты первой волны
Проверить вручную:
- `/`
- `/krovelnye-raboty/`
- `/krovelnye-raboty/remont-krovli/`
- `/krovelnye-raboty/gidroizolyatsiya-krovli/`
- `/krovelnye-raboty/vosstanovlenie-myagkoy-krovli-sinzatim/`
- `/krovelnye-raboty/krovlya-pod-klyuch/`
- `/gidroizolyatsiya/ustranenie-protechek/`
- `/natyazhnye-potolki/pod-klyuch/`
- `/nashi-raboty/`
- `/o-kompanii/`
- `/kontakty/`

Ожидаемо:
- `200 OK`
- страница не пустая
- есть H1
- есть CTA / форма или связка на заявку
- нет явных плейсхолдеров / lorem / пустых карточек

## C. Формы и лидогенерация
### Сценарий 1. Главная форма
1. Отправить тестовую заявку.
2. Проверить response.
3. Проверить создание лида в CRM/webhook-приёмнике.

Ожидаемо:
- success message
- нет `500/501/502`
- лида видно в целевой системе

### Сценарий 2. Форма на странице услуги
Повторить тот же сценарий.

### Сценарий 3. Форма «замер / фото»
Повторить тот же сценарий.

### Негативные проверки
- без имени -> понятная ошибка
- без телефона -> понятная ошибка
- без описания -> понятная ошибка

## D. `/api/leads`
1. Проверить POST с валидным payload.
2. Проверить POST с пустым body.
3. Проверить POST без обязательных полей.
4. Проверить поведение при временной недоступности webhook / CRM.

Ожидаемо:
- валидный payload -> `200 OK`
- невалидный payload -> `400`
- без webhook-конфига не допускается production release

## E. SEO release-smoke
На каждой из ключевых страниц проверить:
- уникальный `<title>`
- meta description
- один H1
- canonical
- index/noindex по ожидаемой логике

Дополнительно:
1. Открыть `/robots.txt`.
2. Открыть `/sitemap.xml`.
3. Проверить, что canonical совпадает с фактическим release URL.
4. Проверить, что служебные / thank-you / тестовые URL не индексируются.

## F. Контент и бренд-контроль
Проверить:
- используется «Крымская Кровельная» в маркетинговой подаче
- официальные блоки не ломают naming
- нет ухода в «универсальную стройку»
- логотип не искажён
- нет псевдокириллицы
- реальные фото / кейсы не заменены пустыми заглушками

## G. Deploy / infra smoke
На VPS проверить:
```bash
docker compose ps
docker compose logs app --tail=200
docker compose logs db --tail=200
```

Ожидаемо:
- `app` и `db` = `Up`
- нет повторяющихся crash loop
- нет ошибок миграций
- нет ошибок подключения к БД

## H. Post-release контроль аналитики и CRM
- тестовый лид дошёл до CRM
- источник / форма / страница передались корректно
- нет дублей из-за повторной отправки
- менеджер видит лид в ожидаемом статусе

## Итоговая фиксация
После smoke теста зафиксировать:
- commit SHA релиза
- дату/время проверки
- кто проверял
- что прошло
- что упало
- решение: GO / rollback / hotfix
